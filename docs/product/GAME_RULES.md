# Jonggu Restaurant 서버 규칙

## 진행 구조

- 하루 루프와 phase 개념은 사용하지 않는다.
- 플레이어는 현재 위치와 자원 상태를 기준으로 탐험, 레시피 선택, 영업을 수행한다.
- 영업과 레시피 선택, 창고 사용은 `Hub`에서만 가능하다.

## 지역

- `Hub`
- `Beach`
- `DeepForest`
- `AbandonedMine`
- `WindHill`

## 자원

- `Fish`
- `Shell`
- `Seaweed`
- `Mushroom`
- `Herb`
- `GlowMoss`
- `WindHerb`

## 도구

- `Rake`
- `FishingRod`
- `Sickle`
- `Lantern`

## 코드값과 표시명

- 서버와 외부 API는 지역, 자원, 도구의 공개 코드값을 그대로 사용한다.
- 현재 서버 정본 코드값은 `FishingRod`, `GlowMoss`, `WindHerb`처럼 공백 없는 식별자다.
- Unity 문서나 화면에서 `Fishing Rod`, `Glow Moss`, `Wind Herb` 같은 표시명을 써도 서버 요청/응답에는 코드값을 사용한다.
- 레시피/재료와 달리 자원/도구는 별도 `resourceCode`, `toolCode` 계약을 유지한다.

## 행동 규칙

- 이동은 현재 지역에서 활성 portal rule이 있어야 가능하다.
- 채집은 현재 지역에서만 가능하고, 필요한 도구가 해금되어 있어야 한다.
- 창고는 `Hub`에서만 사용한다.
- 레시피 선택은 `recipeId` 기준으로 한다.
- 레시피 선택, 영업, 업그레이드 구매는 `Hub`에서만 실행한다.
- 영업은 선택된 레시피와 현재 인벤토리를 기준으로 조리 가능 수를 계산한다.
- 실제 판매 수량은 `min(조리 가능 수, serviceCapacity)`다.

## 재료 시트 규칙

- 재료 정본은 Google Sheets `재료` 시트다.
- 1행은 아래 헤더명을 사용한다.
  - `id`
  - `난이도`
  - `재료명`
  - `수급처`
  - `획득처`
  - `획득방식`
  - `획득도구`
  - `구매가격`
  - `판매가격`
  - `메모`
- 2행부터 `재료명`이 있는 행만 재료로 취급한다.
- `ingredientId`는 `재료` 시트의 `id` 열 값을 서버가 그대로 사용한다. 예: `ingredient_001`, `ingredient_014`

## 레시피 시트 규칙

- 레시피 정본은 Google Sheets `레시피` 시트다.
- 1행은 아래 헤더명을 사용한다.
  - `id`
  - `난이도`
  - `레시피명`
  - `수급처`
  - `조리법`
  - `재료 1`~`재료 7`
  - `가격`
  - `메모`
- 2행부터 `레시피명`이 있는 행만 레시피로 취급한다.
- `recipeId`는 `레시피` 시트의 `id` 열 값을 서버가 그대로 사용한다. 예: `food_001`, `food_041`
- Unity 이미지 파일명도 같은 시트 `id` 값으로 맞춘다.
- `레시피` 시트의 `재료 1`~`재료 7`은 재료명 기준으로 적는다.
- 같은 재료가 여러 칸에 있으면 수량이 누적된다.
- 서버는 `재료` 시트 기준으로 `ingredientId`, `ingredientName`, `quantity` 구조를 만든다.
- 이미지 매핑 기준은 시트 행 순서가 아니라 `id` 열 값이다.
- 시트 재료명은 활성 `ingredients.ingredient_name`과 exact match 해야 한다.

## 운영 규칙

- 도구는 도구 해금 상태로 관리하며 인벤토리 슬롯을 차지하지 않는다.
- 인벤토리는 자원 수량이 아니라 서로 다른 자원 종류 수로 슬롯을 계산한다.
- `AbandonedMine` 진입에는 `Lantern`이 필요하다.
- `WindHillShortcut`은 평판 6 이상이 필요하다.
- 업그레이드는 현재 골드, 자원, 도구, 선행 업그레이드 상태로 판정한다.
