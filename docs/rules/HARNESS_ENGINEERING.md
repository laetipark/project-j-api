# 하네스 엔지니어링 작업 방식

이 저장소는 하네스 엔지니어링 방식으로 운영한다.

## 핵심 원칙

- 에이전트가 읽기 쉬운 저장소를 목표로 한다.
- 큰 단일 규칙 파일 대신 짧은 지도와 주제별 정본 문서를 유지한다.
- 사람은 의도, 제약, 피드백 루프를 설계하고 에이전트는 구현과 검증을 수행한다.
- 문서 구조 자체가 작업 하네스의 일부다.

## 규칙 운영 방식

- `AGENTS.md`, `CLAUDE.md`, `.aiassistant/rules/README.md`는 모두 짧게 유지한다.
- 실제 판단 기준은 `docs/` 아래에 둔다.
- 작업 전에 필요한 문서만 선택해서 읽는다.
- 새 기능이 기존 규칙에 영향을 주면 코드보다 문서를 먼저 갱신한다.

## 작업 루프

1. 관련 문서를 읽어 의도를 좁힌다.
2. 작은 변경 단위로 구현한다.
3. 테스트나 빌드로 바로 검증한다.
4. 배운 내용을 문서에 반영한다.

## 에이전트 친화적 저장소 기준

- 패키지 책임이 명확해야 한다.
- 동적 규칙과 고정 규칙을 분리한다.
- 테이블 주도 값은 코드 상수로 숨기지 않는다.
- 규칙 문서는 검색 가능한 짧은 파일로 나눈다.

## 문서 작성 원칙

- “무엇을 해야 하는가”보다 “어떤 판단으로 유지해야 하는가”를 우선 기록한다.
- 한 문서에는 한 주제를 담는다.
- 세부 예시는 현재 저장소 구조를 기준으로 적는다.
